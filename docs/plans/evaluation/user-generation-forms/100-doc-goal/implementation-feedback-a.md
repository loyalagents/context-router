# Implementation Plan Feedback (A)

- Status: feedback
- Date: 2026-05-21
- Reviews: `implementation-plan.md`
- Read when: deciding whether to execute the 100-doc implementation plan as written

## Overall

This is the strongest plan in the folder and is close to executable as written.
It absorbs the good parts of `a-`/`c-`/`d-` and fixes the two real weaknesses
of `d-`: it splits `d-`'s oversized Batch 1 into two checkpoints, and it
replaces the premature `sdk`/`cli` dual backend with one concrete backend. The
checkpoint structure with per-checkpoint verification commands matches the
repo's plan convention.

The feedback below is mostly tightening, plus one framing issue worth fixing
before execution.

## Verified claims (all accurate)

The plan's concrete repo claims were checked:

- `@google-cloud/vertexai@^1.10.0` **is** a root dependency. The eval scripts
  run from root via `pnpm eval:*`, so the `vertex` backend can import it
  without adding a dependency.
- `apps/backend/src/config/vertex-ai.config.ts` uses exactly `GCP_PROJECT_ID`,
  `VERTEX_REGION`, and `VERTEX_MODEL_ID` (default `gemini-2.5-flash-lite`). The
  plan's env var names match the repo.
- `examples/eval/scripts/run.mjs` supports snapshot updating
  (`compareOrUpdateSnapshots`, `skipExpectedSnapshots: updateSnapshots`), so
  `eval:run --update-snapshots` in Checkpoint 5 is real.

The Vertex backend decision is well-grounded — it reuses infrastructure that
already exists rather than introducing a new SDK and credential path. Better
than `d-`'s SDK default.

## Significant issues

### 1. Be explicit: this batch produces a fixture, not a tested benchmark

The 100 documents are not exercised end-to-end by anything that ships in this
plan. Checkpoint 5's runner hydrates known memory from `profile.yaml` — the
plan says so itself ("Expected runner behavior should match Samir's
known-memory profile, not the document body contents"). So after all seven
checkpoints, the only consumer of the 100 document *bodies* is the validator's
prose linter. The first real consumer — the extraction runner — is Checkpoint
6, which is design-only and deferred.

That sequencing (fixture before consumer) is defensible. The problem is the
framing. The Summary ("200 total realistic corpus documents") and the
acceptance criterion "A known-memory I-9 `filled-form` scenario passes" can
read as "the corpus works." It does not — that scenario would pass with zero
documents, because it never reads them. Recommend:

- State plainly in the Summary that the 100-doc corpus is an unproven fixture
  until Checkpoint 6, and that Checkpoint 5 proves framework plumbing, not
  corpus quality.
- Commit to Checkpoint 6 as the next batch rather than leaving it open-ended
  design, so the corpus does not sit inert.

### 2. `corpus-plan.json` vs `manifest.json` — declare one authoritative

Both files carry per-document `id`, `path`, `category`, `title`, `factKeys`,
`detailTier`, `authority`, `freshness`, and `expectedUse`. The brainstorm
explicitly warned against duplicated per-document data because it drifts. The
plan implies `manifest.json` is generated ("Produce `manifest.json` from the
plan and existing generated body files") — good — but never states the rule.
Make it explicit:

- `corpus-plan.json` is the single authored source.
- `manifest.json` is a deterministic projection generated by `eval:generate`;
  it is never hand-edited.
- The validator asserts `manifest.json` is a faithful projection of
  `corpus-plan.json` (a `MANIFEST_PLAN_MISMATCH`-style check) so the two cannot
  silently diverge.
- Manifest generation must be byte-stable so `git diff` on it is meaningful.

Without that rule you have two hand-editable files with overlapping data — the
exact thing the brainstorm called out.

### 3. Checkpoint 5's snapshot is self-fulfilling unless reviewed

`eval:run --update-snapshots` writes whatever the runner currently produces;
the next `eval:run` then compares against that and passes trivially. So "the
scenario passes" proves nothing unless the generated `expected/filled-form.json`
was reviewed first. Add an explicit step: human-review the snapshot diff before
committing it, and treat that review — not the second `eval:run` — as the
acceptance gate. The repo's templates-scaffold plan and the brainstorm both
stress that snapshot updates must be deliberate.

### 4. Do not inherit the `flash-lite` model default

`EVAL_GENERATION_MODEL` falling back to `VERTEX_MODEL_ID` is sensible, but
`VERTEX_MODEL_ID` defaults to `gemini-2.5-flash-lite` — a small, cheap model
tuned for the backend's product calls. Realistic, varied, non-samey document
prose is the entire point of this initiative; flash-lite will tend to produce
exactly the thin, repetitive output the plan exists to avoid. Recommend the
plan explicitly require a capable model for `EVAL_GENERATION_MODEL` (e.g.
`gemini-2.5-pro`, or `gemini-2.5-flash` at minimum) for the committed corpus,
and not rely on the fallback. Also pin a generation temperature in the script
so reruns are not wildly divergent.

### 5. Some "high-confidence hard checks" have real format variance

Checkpoint 2 lists exact street addresses and full legal names as hard-error
checks from day one. Both vary in real documents: `Avenue`/`Ave`/`Ave.`,
`Apt 12B`/`#12B`/`Unit 12B`; middle name vs middle initial vs omitted. A hard
error on these risks false-positive build failures on legitimately realistic
prose. Recommend the same calibrate-then-promote treatment the plan already
applies to fuzzy checks: street address and full legal name start as warnings
and graduate to errors only after Elena's corpus passes clean. Genuinely
unambiguous identifiers — email, SSN, USCIS/A-number, exact ZIP — are fine as
hard from the start.

## Smaller notes

- **Checkpoint 3 profile expansion ripples.** Expanding
  `samir-desai/profile.yaml` changes the input to
  `seed-preferences.generated.json` and to Samir's existing `template-smoke`
  corpus and its scenario snapshot. The plan claims template-smoke is
  "preserved," but a profile change can break it. Add a step: after any profile
  edit, run `eval:derive-seeds` and re-validate (and if needed re-scaffold)
  `template-smoke` before moving on.
- **`--plan-only` semantics.** Checkpoints 1 and 3 use `--plan-only` but never
  define it. Spell out: it validates `corpus-plan.json` and cross-checks fact
  keys against the profile, and skips document-existence and prose checks.
- **`intentionallyMissing[]` ownership.** The generation prompt includes
  "corpus intentionallyMissing facts," but the plan never says where they are
  authored. State that `intentionallyMissing[]` lives in `corpus-plan.json`.
- **Concurrency.** 100 Vertex calls sequentially is slow; concurrently they can
  hit quota. Add a small, configurable concurrency cap.
- **Context caching.** The per-document prompt has a large shared prefix (rules
  plus profile). Gemini context caching could cut cost, but for a one-time
  100-call run the lifecycle overhead probably is not worth it — fine to skip,
  but say so as a decision rather than leaving it unaddressed.
- **Closeout checkpoint missing.** The `orchestration-plan.md` workflow rule
  requires each batch to end by writing `implementation-summary.md` and
  updating the orchestration status table. Checkpoint 7 captures TODOs but not
  these. Add a closeout checkpoint, and register the 100-doc initiative in
  `orchestration-plan.md` when it starts.

## Bottom line

Execute it, with these adjustments folded in first: fix the framing in issue 1,
make the `corpus-plan`/`manifest` relationship explicit (issue 2), and require
snapshot review (issue 3) — those three change what the plan *claims to
deliver*. Issues 4 and 5 and the smaller notes are quality tuning and can be
handled inside the relevant checkpoints. The checkpoint sequence itself is
sound; do not reorder it.
